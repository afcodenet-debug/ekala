# Public QR Menu – SQLite Replication on Render (Litestream)

## Why this approach (May 2026 decision)

The real `restaurant_tables` rows with the physical QR tokens are generated by the main POS application in its local SQLite database.

The dedicated Render service (`reat-olive-api.onrender.com`) only needs a **near real-time read replica** of that data for the public menu.

**Chosen strategy**:
- Use **Litestream** for continuous, low-latency replication from the writer machine → S3-compatible storage → Render.
- On Render the service runs with `USE_SUPABASE_TABLES=false` and `USE_SUPABASE_PRODUCTS=false`.
- The public menu route (`/menu/table/:qr_token`) uses the replicated local SQLite (already fully supported).
- All the Supabase abstractions we built remain in the codebase for the future migration when you decide to make Supabase the single source of truth.

This gives you working real QR codes in hours instead of weeks.

---

## Architecture

```
Main POS machine (writer)
    data/database.db  ──▶  litestream replicate  ──▶  R2 / S3 bucket

Render public menu service (read replica)
    start-render.sh
        1. litestream restore  →  data/database.db (fresh on every deploy + instance start)
        2. node dist/server/server/server.js
```

Latency: usually < 1 second for new table rows / qr_token updates.

## 1. Create a cheap / free replication bucket

Recommended (2026): **Cloudflare R2** – zero egress fees, generous free tier.

1. Create a bucket (e.g. `great-olive-replica`).
2. Create an R2 API token with read + write permissions.
3. Note the endpoint (looks like `https://<account>.r2.cloudflarestorage.com`).

Alternative: Backblaze B2 or any S3-compatible storage.

---

## 2. On the writer machine (the computer that runs the POS)

Install Litestream (one-time):

```bash
# Linux example
curl -sSL https://github.com/benbjohnson/litestream/releases/download/v0.3.13/litestream-v0.3.13-linux-amd64.tar.gz \
  | tar -xz
sudo mv litestream /usr/local/bin/
```

Create a config file at the root of the project (or `~/.litestream.yml`):

```yaml
# litestream.yml  (place this next to your data/ folder or in $HOME)
dbs:
  - path: ./data/database.db
    replicas:
      - type: s3
        bucket: great-olive-replica          # your bucket name
        path: sqlite-replica                 # prefix inside the bucket
        endpoint: https://<your-account>.r2.cloudflarestorage.com
        access-key-id: ${R2_ACCESS_KEY_ID}
        secret-access-key: ${R2_SECRET_ACCESS_KEY}
```

Run the replicator (as a systemd service, pm2, or simple nohup in production):

```bash
export R2_ACCESS_KEY_ID=...
export R2_SECRET_ACCESS_KEY=...

litestream replicate -config litestream.yml
```

First time it will seed the replica with a full snapshot.

## 3. Render service configuration

In the Render dashboard for `reat-olive-api` (or whatever you named it):

### Environment variables (required)

| Variable                    | Value / Example                                      | Notes |
|-----------------------------|------------------------------------------------------|-------|
| `DATA_DIR`                  | `data`                                               | Where the DB will be restored |
| `USE_SUPABASE_TABLES`       | `false`                                              | Force legacy SQLite path (the replicated DB) |
| `USE_SUPABASE_PRODUCTS`     | `false`                                              | Same |
| `LITESTREAM_REPLICA_URL`    | `s3://great-olive-replica/sqlite-replica`            | Must match the writer config |
| `AWS_ACCESS_KEY_ID`         | your R2 key (or use LITESTREAM_*)                    | Litestream reads these by default |
| `AWS_SECRET_ACCESS_KEY`     | your R2 secret                                       | |
| `AWS_ENDPOINT_URL`          | `https://<account>.r2.cloudflarestorage.com`         | For R2 (optional but recommended) |

The `start-render.sh` script already handles the rest.

### Build & Start commands (Render dashboard)

- **Build Command**: `npm ci --omit=dev && npm run build:render`
- **Start Command**: `npm run start:render`  (already points to `./bin/start-render.sh`)

The script will:
- Download the tiny Litestream binary on first start
- Restore the latest replica (or start empty if none exists yet)
- Launch the Express server

---

## 4. First deployment checklist

1. Push the current code (including `bin/start-render.sh` and the updated `package.json`).
2. Set the environment variables above on Render.
3. Make sure the writer machine is already replicating (step 2).
4. Trigger a manual deploy on Render.
5. In the logs you should see:
   ```
   [Litestream] Restoring from s3://...
   [Litestream] Restore complete
   [Render] Starting Node server...
   ```
6. Test a real QR token: `https://reat-olive-api.onrender.com/menu/table/<real-token>`

You should now get the real table + menu instead of "Table not found".

---

## 5. Monitoring & future migration path

- The diagnostics we added (`[Public Menu]`, `[SupabaseTableRepository]`) remain active and harmless when the flags are false.
- When you are ready to move to Supabase as source of truth:
  1. Change the two `USE_SUPABASE_*` flags to `true` on the Render service.
  2. The menu route will automatically switch to the Supabase repositories.
  3. You can then decommission the Litestream replica for this service.

---

## Troubleshooting

**"Litestream: replica not found"** → First replication from the writer has not completed yet. Wait 30–60 s after starting the replicator on the writer.

**Still seeing SQLite connection on boot?** → Check that the deployment contains the lazy `require()` changes in the two `*repository.provider.ts` files.

**Need live updates while the Render instance is running?**  
Add a background replicate step after restore (advanced). Most public menus are fine with "restore at boot + instance lifetime".
